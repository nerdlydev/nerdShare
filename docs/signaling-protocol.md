# nerdShare — Signaling Protocol Specification

> Wire format, message types, validation rules, and routing for the WebSocket signaling layer.

---

## 1. Transport

| Property       | Value                                    |
| -------------- | ---------------------------------------- |
| Protocol       | WebSocket (RFC 6455)                     |
| Encoding       | JSON (UTF-8 text frames)                 |
| Direction      | Bidirectional                            |
| Authentication | None (MVP) → capability token (Phase 5+) |

### Why JSON Over WebSocket?

- **Human-readable**: Critical during development — `console.log(msg)` tells you everything
- **Adequate performance**: Signaling messages are small (< 5 KB) and infrequent (< 10/sec)
- **Easy validation**: `JSON.parse()` + type narrowing
- **Versionable**: Add fields without breaking existing clients

Binary signaling (e.g., protobuf) is overkill for the message volume and adds a build step.

## 2. Message Type Taxonomy

### Client → Server

| Type            | Purpose                | When Sent                   |
| --------------- | ---------------------- | --------------------------- |
| `JOIN_ROOM`     | Create or join a room  | On page load (host or peer) |
| `OFFER`         | SDP offer for WebRTC   | Host → (via server) → Peer  |
| `ANSWER`        | SDP answer for WebRTC  | Peer → (via server) → Host  |
| `ICE_CANDIDATE` | ICE candidate exchange | During connection setup     |

### Server → Client

| Type            | Purpose                   | When Sent                    |
| --------------- | ------------------------- | ---------------------------- |
| `ROOM_JOINED`   | Confirm room membership   | After successful `JOIN_ROOM` |
| `PEER_JOINED`   | New peer entered the room | On peer join                 |
| `PEER_LEFT`     | Peer disconnected         | On peer WS close             |
| `OFFER`         | Forwarded SDP offer       | On host offer                |
| `ANSWER`        | Forwarded SDP answer      | On peer answer               |
| `ICE_CANDIDATE` | Forwarded ICE candidate   | During ICE gathering         |
| `ERROR`         | Error notification        | On validation failure        |

## 3. Message Schemas

### `JOIN_ROOM` (Client → Server)

```typescript
{
  type: "JOIN_ROOM";
  roomId: string; // UUID v4 — generated by host, extracted from URL by peer
  userId: string; // UUID v4 — generated client-side per session
}
```

**Validation**:

- `roomId` must be non-empty string
- `userId` must be non-empty string
- If room exists: check TTL, check max peers
- If room doesn't exist: create it, assign sender as host

### `ROOM_JOINED` (Server → Client)

```typescript
{
  type: "ROOM_JOINED";
  roomId: string;
  userId: string;
  peers: string[];   // List of existing peer userIds in the room
}
```

### `PEER_JOINED` (Server → Client)

```typescript
{
  type: "PEER_JOINED";
  roomId: string;
  userId: string; // The peer that just joined
}
```

### `PEER_LEFT` (Server → Client)

```typescript
{
  type: "PEER_LEFT";
  roomId: string;
  userId: string; // The peer that left
}
```

### `OFFER` (Relayed via Server)

```typescript
{
  type: "OFFER";
  roomId: string;
  fromUserId: string;
  toUserId: string;
  sdp: RTCSessionDescriptionInit; // { type: "offer", sdp: "v=0..." }
}
```

### `ANSWER` (Relayed via Server)

```typescript
{
  type: "ANSWER";
  roomId: string;
  fromUserId: string;
  toUserId: string;
  sdp: RTCSessionDescriptionInit; // { type: "answer", sdp: "v=0..." }
}
```

### `ICE_CANDIDATE` (Relayed via Server)

```typescript
{
  type: "ICE_CANDIDATE";
  roomId: string;
  fromUserId: string;
  toUserId: string;
  candidate: RTCIceCandidateInit; // { candidate: "...", sdpMid: "...", ... }
}
```

### `ERROR` (Server → Client)

```typescript
{
  type: "ERROR";
  message: string;    // Human-readable error
  code?: string;      // Machine-readable code
}
```

**Error Codes** (MVP):

| Code              | Meaning                          |
| ----------------- | -------------------------------- |
| `ROOM_NOT_FOUND`  | Room ID doesn't exist            |
| `ROOM_EXPIRED`    | Room TTL exceeded                |
| `ROOM_FULL`       | Max peers exceeded               |
| `INVALID_MESSAGE` | Malformed JSON or missing fields |
| `PEER_NOT_FOUND`  | Target peer not in room          |

## 4. Shared Type Definitions

Located in `packages/shared/src/signaling.ts`:

```typescript
export type SignalMessage =
  | { type: "JOIN_ROOM"; roomId: string; userId: string }
  | { type: "ROOM_JOINED"; roomId: string; userId: string; peers: string[] }
  | { type: "PEER_JOINED"; roomId: string; userId: string }
  | { type: "PEER_LEFT"; roomId: string; userId: string }
  | {
      type: "OFFER";
      roomId: string;
      fromUserId: string;
      toUserId: string;
      sdp: RTCSessionDescriptionInit;
    }
  | {
      type: "ANSWER";
      roomId: string;
      fromUserId: string;
      toUserId: string;
      sdp: RTCSessionDescriptionInit;
    }
  | {
      type: "ICE_CANDIDATE";
      roomId: string;
      fromUserId: string;
      toUserId: string;
      candidate: RTCIceCandidateInit;
    }
  | { type: "ERROR"; message: string; code?: string };

export type ClientMessage = Extract<
  SignalMessage,
  { type: "JOIN_ROOM" | "OFFER" | "ANSWER" | "ICE_CANDIDATE" }
>;

export type ServerMessage = Extract<
  SignalMessage,
  {
    type:
      | "ROOM_JOINED"
      | "PEER_JOINED"
      | "PEER_LEFT"
      | "OFFER"
      | "ANSWER"
      | "ICE_CANDIDATE"
      | "ERROR";
  }
>;
```

### Why Discriminated Union?

- **Exhaustive switch**: TypeScript enforces handling every message type
- **No runtime type checking**: `msg.type` narrows the full payload type
- **Refactor-safe**: Add a new message type → compiler flags all unhandled cases

## 5. DataChannel Control Messages (Not Signaling)

These travel over the **DataChannel** (not WebSocket). They are part of the file transfer protocol, not signaling:

| Message             | Direction   | Payload                                                            |
| ------------------- | ----------- | ------------------------------------------------------------------ |
| `FILE_META`         | Host → Peer | `{ fileId, fileName, fileSize, mimeType, chunkSize, totalChunks }` |
| `HELLO_ACK`         | Peer → Host | `{ fileId }`                                                       |
| `TRANSFER_COMPLETE` | Host → Peer | `{ fileId }`                                                       |
| `ERROR`             | Either      | `{ message }`                                                      |

```typescript
// DataChannel control messages (separate from signaling)
export type DCControlMessage =
  | {
      type: "FILE_META";
      fileId: string;
      fileName: string;
      fileSize: number;
      mimeType: string;
      chunkSize: number;
      totalChunks: number;
    }
  | { type: "HELLO_ACK"; fileId: string }
  | { type: "TRANSFER_COMPLETE"; fileId: string }
  | { type: "DC_ERROR"; message: string };
```

## 6. Server Routing Logic

### Message Dispatch

```typescript
function handleMessage(ws: WebSocket, raw: string) {
  let msg: ClientMessage;
  try {
    msg = JSON.parse(raw);
  } catch {
    ws.send(
      JSON.stringify({
        type: "ERROR",
        message: "Invalid JSON",
        code: "INVALID_MESSAGE",
      }),
    );
    return;
  }

  if (!msg.type || !msg.roomId) {
    ws.send(
      JSON.stringify({
        type: "ERROR",
        message: "Missing required fields",
        code: "INVALID_MESSAGE",
      }),
    );
    return;
  }

  switch (msg.type) {
    case "JOIN_ROOM":
      handleJoinRoom(ws, msg);
      break;
    case "OFFER":
    case "ANSWER":
    case "ICE_CANDIDATE":
      relayToTarget(msg);
      break;
    default:
      ws.send(
        JSON.stringify({
          type: "ERROR",
          message: "Unknown message type",
          code: "INVALID_MESSAGE",
        }),
      );
  }
}
```

### Targeted Relay

```typescript
function relayToTarget(msg: ClientMessage & { toUserId: string }) {
  const room = rooms.get(msg.roomId);
  if (!room) return;

  const targetWs =
    room.peers.get(msg.toUserId) ??
    (room.hostId === msg.toUserId ? room.hostWs : null);
  if (targetWs) {
    targetWs.send(JSON.stringify(msg));
  }
}
```

## 7. Sequence: Full Signaling Handshake

```
Host Browser          Signaling Server        Peer Browser
     │                      │                      │
     │── WS connect ───────►│                      │
     │── JOIN_ROOM ────────►│                      │
     │◄── ROOM_JOINED ─────│                      │
     │   (peers: [])        │                      │
     │                      │◄──── WS connect ─────│
     │                      │◄──── JOIN_ROOM ──────│
     │                      │───── ROOM_JOINED ───►│
     │                      │      (peers: [host])  │
     │◄── PEER_JOINED ─────│                      │
     │                      │                      │
     │── OFFER ────────────►│───── OFFER ─────────►│
     │                      │◄──── ANSWER ────────│
     │◄── ANSWER ───────────│                      │
     │                      │                      │
     │── ICE_CANDIDATE ────►│──── ICE_CANDIDATE ──►│
     │◄── ICE_CANDIDATE ────│◄─── ICE_CANDIDATE ──│
     │   (trickle, multiple rounds)                │
     │                      │                      │
     │◄═══════════ WebRTC DataChannel opens ══════►│
     │        (signaling server no longer needed    │
     │         for data transfer)                   │
```
